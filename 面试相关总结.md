- **高精度混合 RAG 系统**：针对运维领域专有名词多、长尾问题显著的特点，设计了 **Hybrid Search** 混合检索链路。融合 **BM25** (稀疏检索，精准匹配错误码) 与 **FAISS** (稠密向量检索，语义匹配) 结果，应用 **RRF (Reciprocal Rank Fusion)** 倒数秩融合算法优化排序，有效解决了单一向量检索在低频术语上的召回缺陷。实现基于 Markdown Header 的文档切分策略，并在后处理阶段引入 **Re-ranker** (基于 Cosine Similarity) 精排，大幅抑制了“Lost in the Middle”现象。
- **AIOps 智能诊断闭环**：基于 **Plan-Execute-Replan** 思想实现了全自动故障诊断逻辑。Agent 能够并行调度 **Log Tool** (日志摘要分析)、**Prometheus Tool** (指标异常检测) 及 **RAG Tool**，综合关联多模态运维数据生成 RCA 报告。对于信息不足场景，实现了基于 **Replan Node** 的动态追问与补充查询机制，能够自主修正推理路径，将复杂故障的排查时间从小时级缩短至分钟级。
- **对话功能与工程化落地**：基于 **FastAPI** 构建高并发后端服务，利用 **SSE (Server-Sent Events)** 技术实现 Agent 推理过程的实时流式输出，解决大模型响应延迟带来的体验问题。设计了 **LLM Provider 适配层**，支持 OpenAI、混元等模型的无缝热切换，保证系统的模型无关性。集成 **LangFuse** 全链路可观测性平台，实现了从 Router 到 Tool Call 的完整 Trace 追踪与 Token 成本审计，为 Prompt 调优与性能瓶颈定位提供了可视化数据支撑。

### 如果让你重新设计，你会做出什么改进与优化 / 项目还有哪些不足，如何改进

由于先前的系统主要是在本地开发使用的，其稳定性和扩展性还存在一定缺陷，并不是非常适配企业的实际应用。如果以真正的工业级标准来看，我认为在以下三个维度还有明显的演进空间：

1. **从“单机有状态”向“分布式无状态”演进（状态与存储的解耦）**
   - **现状与痛点**：目前系统如果遇到服务重启的情况，用户的上下文容易丢失，同时我的RAG索引也需要重建。这是因为目前本地开发基本都是内存存储，图的 Checkpoint 状态记忆用的是基于内存的存储器 `MemorySaver`，RAG 用的是单机版 FAISS。如果遇到服务重启等异常情况，用户的上下文就会由于 session 不共享而彻底丢失。这样的设计符合当下的小系统，速度快且便于迭代开发，但是不符合实际生产。
   
   - **架构优化方案**：
   
     由于 LangGraph 的解耦特性，只用将 MemorySaver 替换为 `RedisSaver` 或 `PostgresSaver`并连接数据库，就可以实现 API 计算节点的无状态化；
   
     同时将RAG模块的 单机 FAISS 升级成 **Milvus** 等分布式向量数据库，从而具备海量文档承载能力，服务重启也不会丢失数据。同时，FAISS插入和删除文档还常常要重建索引，如果引入Milvus则可以实现动态增删数据。此外Milvus还支持结构化过滤+向量检索，比如”只查最近一天“的数据等等
   
     【checkpoint——langgraph的状态快照，每一步执行后，进入新的节点之前都会保存一个state，让agent能够从这个断点继续跑】
   
2. **解决日志截断问题**
   
   - **现状与痛点**：目前在获取到日志之后，我只是新开一个LLM让它对其进行脱水和总结，保留error和时间戳，id之类的信息，一旦日志过长不但会消耗大量token，还容易导致关键报错被抛弃或者修改，对话或者运维agent可能出现幻觉导致诊断错误，但如果日志完全传给运维agent组装进他的上下文，又可能导致模型注意力丢失。
   - **架构优化方案**：在海量的日志数据传给到agent推理之前，可以先用正则匹配来保留最关键的字段，并且引入一个本地部署的、轻量级的**小模型（如 专门微调过的 Qwen2.5-7B）** ，低成本地对前面的关键内容继续压缩，只把精炼后的**报错摘要**抛给后端或云端推理大模型做 根因归因（RCA）推理。这样既保证了质量，又降低了 API 计费成本。
   
3. **从“感性测试”走向“自动化数据驱动”的 LLMOps 评估体系**
   - **现状与痛点**：目前对于 Prompt 的修改 以及 RAG 检索中的 top-K 和 置信度阈值调参，主要是依赖我手动构造的几十个告警 Case 来做人工检查，这种方式数据量少且带有主观性，既没有说服力，也难以保证agent能在更广泛的测试集上展现出更好的效果。
   - **架构优化方案**：所以应该建设一个标准的 **LLMOps 自动化评估体系**。将评测接入 CI/CD 流水线。每次修改 Prompt 或 RAG 参数（例如 top-K、阈值）后，自动在测试集上运行评估，根据结果来看是否要合并到main或者origin上。评估时主要关注三个核心指标：
   - Context Precision上下文精确度，用于衡量检索结果有多少比例是与问题相关的内容
   - Faithfulness幻觉检测，评估模型回答是否严格基于检索的内容，而不是胡编的
   - Answer Relevance回答准确度，用于判断回答是否真正解决用户问题，这个可以用一些人工标注的ground truth数据集

### 简单说说LangGraph是什么
 **LangGraph 是 LangChain 生态中的一个 Agent 工作流框架，用来构建基于状态图（StateGraph）的 AI Agent 系统。**

 和传统的 AgentExecutor 不同，它允许开发者把 Agent 的执行流程建模成一个图结构，每个节点可以是一个 LLM 推理步骤、工具调用或者普通函数。图结构负责控制执行路径，比如分支、循环、并行、状态管理。而每个节点内部可以调用 LLM 来进行推理或决策。

因此LangGraph能够把Agent从黑盒循环变成一个可控的工作流，提高稳定性和开发效率，更适合企业落地生产。

（比起传统workflow就是加了个结点内的LLM推理）
（比起传统AgentExecutor就是加了个可以更好控制的图结构）


### 什么是Agent
agent是一个能够感知环境、自主决策并执行动作的智能系统。它通过大预言模型作为核心推理引擎，结合工具调用、记忆机制和规划能力来完成复杂任务。有点像是人类与大模型之间的一个中间人，它负责接收用户指令，并协调AI和实际工具来干活。

对比传统程序：

本质区别在于执行逻辑不同，传统程序遵循预定义的代码逻辑，而Agent具备自主推理能力，会根据当前状态动态生成执行计划，即使是同一个输入，在不同上下文下可能产生不同处理策略。

“接受不确定性”

如果要结合项目：

举个简单的例子，比如我们给agent准备好一些基本工具，如查找文件、读取文件、移动文件等。当用户发出指令，比如帮我读取C盘目录下的hello_world.cpp文件，移动到D盘目录下，那么那么整个流程大致会按照下面的步骤执行：

1. 智能体agent会先把这个请求传给 AI ，并附带告诉 AI 它可以使用哪些工具，工具有哪些作用。AI经过思考后，会告诉智能体agent:调用移动文件工具
2. agent收到指示后，就实际操作工具移动文件，AI收到移动完成的进度后，返回总结内容给agent，agent向用户报告结果。
因此，agent让AI变成了能动手的实干家，过程更加自动化智能化。

### agent是怎么实现工具调用的
在没有function calling技术之前，工具描述是放在系统提示词当中的，agent会通过返回字符串的方式调用工具，但是这种方式容易出错。
（比如告诉大模型有一个查询天气的工具，参数是城市和日期
大模型返回字符串可能是:调用工具:查询天气的工具，输入上海，明天然后我们后端解析字符串，解析到了要调用天气工具但是参数错了!明天不是一个日期!）

现在有了function calling之后，大模型就具备了“按格式调用函数”的能力。我们就可以用JSON格式描述我们写的函数名，函数介绍，参数字段，并规范AI调用工具的回复格式，然后通过调用AI接口得到AI的JSON回复。我的项目就是用这种方式调用工具的。
（还有些模型的接口会给出专门的规范化的tools字段，用来让我们传入定义。）
开发者不用自己写代码检测AI回复是否正确，若AI回复错误，AI的服务器端可检测并自动重试，降低用户端开发难度和token开销。

然后是运维Agent，它是为了解决值班排查问题的痛点而做的，传统的处理方式往往依赖于人工排查：看告警查日志然后才能分析问题，尤其是在晚上或节假日，告警一多就容易影响判断速度和准确性。运维Agent通过调用多平台的api，一站式完成问题排查。例如：大模型能自动从告警中提取接口名和时间范围等，查询日志和告警处理手册，将信息全部汇总输出并给出一个根因分析报告，并且在操作风险较低时自动解决问题。
【从实际架构上来说，运维Agent 是通过LangGraph将多个Agent和工具编排起来的，见Q2】

### 什么是ReAct/ LangChain的Agent执行器是怎么工作的
- ReAct的本质是让大模型边推理边行动，先思考问题然后通过调用工具来处理任务，每一步都依赖于上一步，优势是让Agent能实时决策，动态调用工具。

- LangChain的agent执行器采用的就是ReACt的模型，通过推理，行动、观察三个环节的循环迭代解决复杂问题。
  - 执行器接收用户输入后，Agent会分析当前状态并决定下一步行动，这个决策过程依赖于内置的推理逻辑和预定义的工具集。
  - 行动时，Agent可以选择使用特定工具(如搜索引擎、API调用、数据库查询)或直接给出最终答案。
  - 工具执行完成后，Agent会观察返回结果，然后基于新信息重新评估情况，决定是否需要使用其他工具或已经可以回答用户问题。这个循环会持续进行，直到Agent认为已经收集到足够信息给出满意答案，或者达到预设的最大迭代次数。
    整个过程中，执行器负责协调Agent的推理逻辑、工具调用和结果整合，确保每个步骤都能正确执行并传递给下一个环节。
  - 执行器还会处理异常情况，比如工具调用失败、超时或返回错误格式数据时的容错机制保证整个推理链路的稳定性。

### 对话Agent的使用场景是什么
主要有3个场景：
1、业务场景：上游的业务同事可能经常回来询问这个API为什么报错，接口是怎么接的等等，有了对话Agent之后，相关问题的响应时间就可以从分钟变为秒回，只要文档写的够清楚就不用被重复询问
2、值班场景：收到告警之后，每次都去查看手册会比较麻烦，现在直接把告警信息发给agent，它就能快速检索告警处理手册中的标准处理方法，并结合实际情况给出分析。
3、工单预处理场景：对话Agent未来还可以对接工单平台，绝大多数工单都是简单或者重复问题，历史工单记录如果被存储进知识库，对话Agent就可以召回并根据此进行自动回复，只把复杂问题交由人工解决。

### 运维agent/多agent怎么协作

在我的项目中，我设计了两套Agent系统：对话Agent和运维多Agent系统。对话Agent采用ReAct的单Agent架构，而运维agent则是基于规划-执行-反思 的设计模式，构建了一个多Agent协作框架。

多Agent协作的核心是通过图结构来组织不同Agent的职责分工和状态流转。具体来说，我将系统拆分为多个功能Agent，包括意图识别Agent、指标分析Agent、日志分析Agent、RAG检索Agent、根因分析Agent、反思Agent以及风险控制Agent。

各个Agent通过共享全局状态进行通信，每个节点在执行后会更新状态并传递给下一个节点，整体流程是一个带有条件分支和循环的有向图。

在第一阶段，由意图识别 Agent 对用户输入进行解析，对当前任务做出规划，决定后续的执行路径。

在第二阶段，是信息获取阶段，我设计了三个并行的分析 Agent，包括日志分析、指标分析和 RAG 检索 Agent。这些 Agent 可以并行执行，从不同数据源获取信息，并将结果写入共享状态。

第三阶段是决策阶段，由根因分析 Agent 对多源信息进行综合分析，给出问题定位和处理建议。

第四是阶段，如果当前信息不足以支撑结论，则会进入反思 Agent，由其判断缺失的信息类型，并重新调度相应的分析 Agent，从而形成一个闭环的自我修正流程，反思的次数限制在三轮，超过三轮会直接告知用户哪些信息不足，需要补充。

在最终阶段，还会引入风险分级Agent，对操作进行安全评估，高风险操作需要经过人工审批后再执行，从而保证系统的安全性和可靠性。

### 运维Agent的使用场景

它主要是为了解决值班排查问题的痛点而做的。当服务非常多的时候，服务错误、性能波动、中间件异常、下游故障依赖等等告警数量太多，同时日志、监控、告警等信息分散，对值班工作的效率和准确性产生较大影响。
而运维Agent主要使用了Plan-Execute-Replan的设计模式，自动规划排查步骤，通过工具调用个平台API，一站式完成排查

### 什么是Plan-Execute-Replan的设计模式【Agent的反思机制】
是Agent的结构化任务执行模式，先规划执行步骤，制定完整计划，再按照步骤行动，接着评估结果，决定是否校准方向。这种方式能让agent像人类一样拆解复杂任务，还能应对突发变化。
这个模式靠三 **类** 子Agent协同：（与Q2基本一致）
1、规划Agent：负责把任务拆解生成清晰步骤
2、执行Agent：调用工具执行步骤，推进任务
3、评估或者说重规划Agent：评估执行结果，有效则执行下一步，异常则重新调整。

核心优势是让复杂的任务变得可控又灵活，把复杂任务拆成可执行的步骤，降低认知符合，评估机制又能及时调整计划，避免大模型幻觉导致严重后果。

【如果追问详细内容或者要求结合项目解释，见Q2——三类agent怎么协作】

【与ReAct的区别：这个先规划了整体步骤再去执行，并且会评估。ReAct则是让大模型一边推理一边调用工具，没有固定步骤，没有全局规划。比如react会先查日志，然后根据这个结果再去决定要不要去调用其他工具，而plan则是一开始就规划好用什么，最后去反思。
所以我的项目中对话场景用ReAct，因为用户问题比较灵活，需要实时决策，比如回答某个API怎么使用等等，这些任务的执行链条不适合预设太长。而运维agent的职责就是接收一个明确目标，然后指定一个可能包含多步骤的排查计划，严格按计划调用工具执行】

### Langfuse是什么，有什么作用，为什么要实现，怎么实现
首先，Langfuse是一个专门设计给大模型，特别是Agent的开源可观测平台。
在多Agent和ReAct工作流中，如果我们不去对中间过程的大模型输出进行监控，那么在最终报告出现错误的时候，我们就根本不知道到底是路由错误、RAG系统没有成功召回，还是LLM出现幻觉乱写导致的。同时，我们也不知道到底是哪些节点耗时过长，或者是消耗的Token过多，导致整个系统太费时费钱。

而接入Langfuse之后，Agent的内部步骤就会更加清晰，方便我定位错误、减少消耗和调整架构。

具体的实现方式是利用Langfuse相关库提供的Handler，在图执行的入口，注册这个处理器，并且把每个对话的id装配进去即可。由于LangGraph框架本身支持这个接口约定，所以我后面就可以在Langfuse平台上看到每个对话的内部详细过程。

**生产与开发的区别**：用docker部署可以实现离线使用Langfuse。

### 遇到过什么困难/挑战
在开发过程中，普通的 Bug 其实很好解决，但真的让我卡了很久，并深刻反思架构设计的，主要是面对大模型本身的不确定性时，Agent系统该怎么去给他兜底的问题。
比如在开发 Chat Agent 的 ReAct 循环时，发现当知识库召回的内容过长，或者某个工具（如RAG检索召回）返回的数据质量不高时，大模型会陷入“无限思考并盲目调用工具”的死循环，最终耗尽 Token 报错。
我通过接入 LangFuse 观察 Trace 链路，发现原因是大模型在中间过程中出现了注意力丢失。由于数据太长且内容质量低，大模型忘记了最初的任务目标，只是在根据最近一次工具的错误返回瞎试。于是我在RAG系统的重排结尾设定了一个 相似得分阈值，一旦某些文档块得分低于这个阈值，就会被过滤不进入重排。同时我又在日志查询工具限制了返回长度，进行了适当的截断，这两步减轻了大模型的认知负荷。
然后我又限制了ReAct流程的轮数为5轮，避免无限思考与无限调用工具，一旦5轮没有出来，大模型就会被强制路由到“信息不足节点”，告诉我们它缺少什么信息，要求人类补充。
最终系统响应成功率从最初极不稳定的 70% 提升并稳定在 95% 以上，接口请求耗时和 Token 成本大幅下降。

#### 回答方案二：主打工程化与系统稳定性（高危操作拦截与防大模型幻觉）
**挑战描述**：大模型在生成运维操作指令时具有不可控性（幻觉），可能会生成如“重启核心数据库”这样的高危指令，直接让其自动化执行风险极高。
**解决过程 (Action)**：
1. **初期尝试（失败）**：一开始我试图只通过精心编写 Prompt 来严令禁止大模型返回高危操作，但发现面对长篇复杂的告警日志时，模型注意力会分散，仍然存在极小概率的违规操作。
2. **算法层兜底（Regex + LLM双重检验）**：我决定在执行前加入一个专门的 `action_router_node`。第一层走代码写死的正则库匹配敏感词；第二层利用一个独立的小模型进行意图语义审核，如果是高危行为直接打回。
3. **架构层兜底（引入 HITL）**：即便如此我依然觉得不够安全。于是我深挖了 LangGraph 的底层机制，利用了它的 `Checkpointer` (记忆检查点) 和 `interrupt_before` 机制。把整个运行状态流冻结并序列化到内存中，抛出后端等待状态。前端人工审批通过后，再拿着原来的 `Thread_id` 恢复系统流转。
**最终结果 (Result)**：彻底终结了 AI 自动化运维中的“盲盒执行”风险，将一个玩具级别的 Agent 提升到了具备生产落地可行性的工业级系统。

### Q1：为什么选择 LangGraph 构建工作流？而不是用传统的 LangChain AgentExecutor 或者手写 Prompt 编排？
**标准回答**：
如果不使用LangGraph而是只用LangChain的AgentExecutor进行固定的 ReAct 单循环，大模型就会自己决定调用什么工具、什么时候结束，所有中间状态通过 Prompt 拼接传递，这只适合用于简单聊天场景且会额外消耗tokens，而在运维排障这种的复杂业务流转中，流程最好是严格控制，先读取告警、再去查询日志、然后根据知识库内容生成报告，不适合让大模型太自由地发挥；手写编排虽然灵活，但维护状态和上下文非常繁琐。
而引入 LangGraph 的 StateGraph 可以解决三个痛点：

1. **全局状态管理（State）**：通过定义 运维Agent的共享状态`OpsState`（TypedDict），像日志总结`log_summary`、`metrics_summary` 之类的独立步骤产物可以安全地跨节点流转，不用在 Prompt 里传来传去。比如日志工具得到了5000行原始数据，他把原始数据总结完之后放到state中的日志摘要字段即可，其他llm不需要读取
2. **图结构的编排能力**：支持循环（Replan 追问）、条件分支（Action Router 风险分级）和并发执行（日志/指标/RAG 三路并发），将复杂的 RCA 诊断拆解为清晰的图节点。
3. **断点与人为干预（HITL）**：大模型的执行在故障处理中有引发二次故障的风险，LangGraph 原生支持 Checkpoint，能在执行高危操作的节点处中断，等待人工确再恢复，这是传统 Agent 框架极难做到的。

【并发怎么实现】 图框架 add_edge 自动并发，我设置一个router路由节点并连接了日志收集、指标分析和RAG知识库三个节点，他们之间没有相互依赖，会自动并发收集数据
【条件分支和循环 / 信息不足追问 怎么实现？】添加一个条件边，让AI Agent进行诊断，判断是否出现信息不足的情况，有则会流转回 追问replan节点回头重新查询/询问

### Q2：你的 Ops 工作流中为什么设计多个 Agent（Router, Log, Metrics, Diagnosis）而不是用一个 Agent 挂载所有工具？怎么设计的
**标准回答**：
“全能型 Agent” 在实际工程中往往表现极差。把所有工具（查日志、查指标、查知识库等）都挂载到一个大模型上，不仅会导致 Prompt 极长、上下文容易溢出，还会显著增加模型规划路径的难度和发生“幻觉”调用错工具的概率。
采用**多 Agent 协作（Multi-Agent）架构**是为了实现**关注点分离（SoC）**：（下面是怎么实现/设计的）

1. 首先是规划Agent：是整个运维agent的入口，拿到告警信息和用户的补充描述后，专门做信息识别和任务拆解规划，决定要调用哪些工具。
2. 然后是各种领域专家（**Log日志分析 / Metrics指标专家 /RAG检索器**）来执行步骤 ，比如 日志分析专家只关心Error堆栈提取（通过日志tools收集然后给llm分析），Metrics Agent 的 Prompt 专门被优化为去提取 PromQL 和阈值，RAG检索器把用户描述和告警信息组合成检索文本然后召回，它们只负责调用工具来执行规划Agent的步骤。（三个并发进行信息采集）
3. 紧接着是 根因分析**Diagnosis Agent** ，前面的各种专家会把信息汇总到状态记忆State中，而他则会基于这些汇总摘要进行 RCA（根因分析）逻辑推理，产出报告。同时当报告中出现信息不足等字样的时候，会自动进入重问环节，再次让对应专家收集信息。
这种架构缩小了单个 Agent 的解题空间，大幅提升了诊断输出的准确率和稳定性，避免了agent间直接对话可能导致的不可控情况。

**深层追问**：如果有新的一种数据源（比如 Tracing 链路追踪数据）加入，整个系统怎么做最小代价扩展？

### Q3：为什么要实现人工审批？高风险操作的人工审批闭环（HITL）在代码逻辑上是如何落地的？
**标准回答**：
原因：
大模型完全自动化执行所有操作的风险很高，我们无法确定agent输出的报告和解决方式是完全正确的，如果要执行重启服务、扩容等高风险操作，人工审批确认会是更好的选择。

实现思路主要分为三步：
1. 在根因分析agent生成报告后，系统先会进入一个风险分级的条件边中，如果检查到报告中出现了任何预设的高风险操作的词语，就会标记为高风险操作，如果不是高风险操作，还会再使用大模型作为辅助判断，二次确认操作的风险程度，若仍然是低风险操作，那么就跳过人类审批。
2. 若确认为高风险操作，就进入人工审批结点，这个结点在构建StateGraph 时就已经被设为挂起点（通过 `compile(interrupt_before=["human_approval"])`），流程不会继续执行，而是暂停等待 【NodeInterrupt异常（LangGraph自带）】。
3. LangGraph的checkpoint机制可以把当前的状态保存下来，直到后端API收到主管的审批通过信息，把approve=True写入全局状态，工作流才会继续。 
（如果深问后续的操作：agent会把报告最后得出的操作建议 封装成结构化任务【包含操作类型、目标服务、参数等】，并且交给执行节点，执行节点可以调用真实的运维API或者自动化平台，进行类似重启容器/服务回滚/容量扩展/清理缓存等操作。在当前项目中，由于没有真实运维环境，我主要实现的是流程和接口设计，具体操作部分使用模拟工具返回结果，但系统架构已经预留了真实执行接口。）

## 【RAG 设计】

### Q4: RAG是什么，怎么实现的
RAG是构建智能客服、企业知识库等Agent项目的核心技术，中文全称是检索增强生成。
当我们需要让大模型回答特定领域问题（如内部文档）的时候，直接将全部长文本发给模型会受限与模型上下文窗口大小，导致成本高，速度慢，准确率低。
RAG通过先检索相关内容再生成答案的方式，完美解决了这些问题，在对话Agent和运维Agent里，都会使用到RAG。
具体流程是:用户提出问题后，系统先从知识库检索相关的真实信息，再把这些检索到的可靠知识和问题一起输入给生成模型，让模型基于这些有据可依的内容生成回答。这样输出的结果不仅更准确，还能追溯信息来源，也能覆盖模型训练数据之外的新知识，减少大模型幻觉的同时还能减少token消耗。

RAG的核心流程分为两部分
提问前(离线处理，数据准备)分为3个步骤:文档分片->embedding向量化 ->存储
提问后(在线处理，回答生成)分为3个步骤:召回->重排 -> 生成

【需要我更详细地介绍一下每个步骤吗：】
提问前链路/导入知识库文档：将原始文档切割为多个**语义完整**的片段，每个片段聚焦一个具体功能或者问题。然后是索引——用Embedding模型将每个片段转为向量，语义相近的文本向量距离更近。最后将**片段文本和向量**存入向量数据库。
（向量数据库不仅存储向量，还保留原始文本，因为最终生成答案需要的是文本内容，向量只是用于相似度计算。）

提问后链路/回答生成：
召回：从用向量相似度算法计算问题向量与数据库中所有片段向量的相似度，挑出Top N(如10个)最相关的片段。速度快、成本低，但准确率有限，适合初步筛选。

重排:给片段排优先级
召回的10个片段可能仍有冗余或相关性不足，需要进一步精筛:
1.使用专门计算文本对相似度的cross-encoder模型，逐对计算用户问题与每个召回片段的语义相关性。同时，如果相似度得分低于设置的阈值，也就是相似度不够高，那么这些文档块就会被过滤，不会被传输给LLM。
（具体就是将待匹配的关键prompt和每个文档块各拼接成一个输入，传输给cross-encoder模型，最后输出一个相似性得分，我们只需要排序后拿出topk比如3个最高分的即可）
2.最终排名选出TopK(如3个)最相关的片段。

生成:让AI写出答案
将重排后的3个相关片段与用户问题一起投喂给大模型，模型基于这些片段生成准确、简洁的回答。
优势:避免模型幻觉，答案严格基于检索到的事实性内容，且输入内容少(仅3个片段)，成本和速度都更优。

### Q4.1：为什么使用RAG而不是微调大模型

在这个智能运维助手项目中，我选择 **RAG而不是微调大模型**，主要有三个原因：

第一，**运维知识更新非常频繁**，比如新的故障案例、排障流程、Prometheus指标说明等。如果使用微调，每次知识更新都需要重新训练模型，而RAG只需要更新知识库即可。

第二，**RAG可以保证知识来源可追溯**。在回答问题时，系统会检索相关文档并注入到Prompt中，模型生成的回答就能明确指出信息来源于哪些运维文档，这能帮助我们检查Agent所说的内容是否准确。

第三，**成本更低、实现更简单**。微调需要GPU资源和训练流程，而RAG只需要Embedding和向量检索即可实现知识增强。

因此对于这种 **知识密集型问答系统**，RAG是更合适的方案。

**微调：** 微调更适合解决模型能力问题，RAG主要解决知识问题。例如，微调可以改变模型输出风格，学会特定任务格式，提升某类任务能力，但这个项目LLM已经有很强推理能力。当然，将RAG与微调结合，能让agent在垂直领域起到更好的效果。

### Q4.2 FAISS和BM25的双路召回和融合是怎么实现的

单纯的向量召回（FAISS）有时会对专有名词不敏感的问题，而 BM25 擅长关键词匹配。双路召回是为了结合 **语义理解** 和 **精确匹配**，提升长尾问题的召回率。

在离线建库的时候，除了文档块的向量化存储，我还调用BM25的api构建并且存储了倒排索引，这个索引包括了每个词出现在了哪些文档中，这些文档的长度是多少等等。在召回的时候，这些统计量可以帮助我们计算BM25的得分，在得到了另一路的向量相似度得分之后，会使用RRF算法进行融合。这个RRF算法主要是关注两路召回得到的文档块的排名，只有两路得分排名都较高的文档块才会被召回，进入重排阶段。（1除以每路召回的排名和，分母还有个k）

### Q5.1：（召回中用到的）FAISS （读音像faice）是什么，怎么实现的
FAISS 是 Meta 开源的高性能**向量相似度搜索库**，用于在大规模向量集合中快速找到与 query 向量最相似的 Top-K 向量，常用于 RAG、推荐系统和语义搜索。（是一种库，而不是数据库）
在 RAG 系统中，我们通常会先使用 embedding 模型把文档块 转换成向量，然后把这些向量构建成 FAISS 索引，我这里是存储在了内存中，如果是工业上可以存在milvus。
索引本身只存储向量数据，在查询时，我们会把 查询文本 转换为 向量，然后在 FAISS 索引中查找最相似的向量，FAISS 会返回对应的向量 ID。
在应用层（例如 LangChain）会维护一个向量 ID 到原始文档块 的映射，通过这个映射就可以找到对应的文本内容。
而在这个项目中，FAISS构建的向量索引（<向量id，向量>）和LangChain维护的映射会被存在本地，避免数据丢失。

### Q5.2：为什么工业 RAG 很少直接用 FAISS，而是用 Milvus /ˈmɪlvəs/
因为FAISS是向量搜索库而不是数据库，它缺少分布式能力，文档块多的时候内存放不下。
此外，FAISS删除数据很困难，**向量id必须对应行号**，因此删除数据很多时候都要重建索引，而工业数据库则不需要。
还有就是它不支持并发控制、不提供API和服务，需要自己实现等等。

类似问题**如果在生产环境数据量千万级别，你的FAISS扛得住吗**：不行，原因见上，缺少分布式能力，单机内存不够等。要从FAISS转变为使用Milvus也不困难，只需要换成这些分布式向量数据库的SDK即可

### Q5.3：如何处理文档更新，增量索引是怎么做的
**标准回答**：
这个本质上就是数据库的增删改查。当文档更新时，系统会给每篇文档甚至每个被切分出来的文档块（Chunk）分配一个全局唯一的 ID（通常是把文档名加内容做个哈希算出来的）。
如果新增一篇文档，就直接走一遍切分、向量化然后插入向量库对应的集合里。
如果是修改或删除老文档，就拿着之前生成的那个文档 ID，去向量库里用 Delete 接口删掉旧的文档块即可，然后再把修改后的文档以同样的流程当作新文档存进去。

**防深究话术：**
“其实目前我们项目的文档库体量不大（比如也就是几百份运维手册），如果上全套按句子的流式增量更新反而大材小用。所以工程上最稳妥的做法就是**直接基于文档 ID 进行覆盖（先删后增）**。对于变化不频繁的数据，偶尔晚上写个定时脚本做一把全量构建也就两三分钟的事，足够满足生产环境了。”


### Q6：Embedding是什么意思
向量化，通过大模型接口把文本变成了一堆数字组成的向量。
为什么：语义相近的句子，转化后他们在数学空间里的距离就很近。只要计算向量之间夹角（余弦相似度），夹角越小余弦值越靠近1，就代表语义越接近。能帮我们快速找出现有知识库里最相关的段落。


### Q6.1：BM25是什么？为什么要用混合检索（双路召回）？
**BM25 是什么**：
BM25 是一种传统的、基于“关键词匹配”和“词频”的粗排检索算法（TF-IDF 的升级版）。简单来说，它不理解句子的涵义，而是直接拿着你搜索的关键词去整个文档库里找包含这个词的文本，这个词在整个知识库中越生僻、在这段文本的出现次数越多，那这段文本得分就越高。

**双路召回是怎么实现的/为什么用双路**：
因为两者是强烈互补的。
纯向量检索（FAISS）擅长的是**语义理解**，比如搜“服务器崩了”，它能通过模型“意会”出我们要找说明“宕机”的内容。但向量模型有个致命痛点，就是对**运维特有名词、长尾错误码、特定的 Pod 服务名称**（比如 `OOMKilled`、`NullPointerException`、`nginx-ingress-5g7bz`）完全不敏感，可能会胡乱匹配一些语义相近但并不相关的故障单。
而 **BM25 则极度擅长这种确切标识符的字面精准匹配**。所以在具体实现上，每次查询到来，我会将请求同时分发给 BM25 库和 FAISS 库，各自取回最相关的 Top-N 个文档块进入候选池。这样既保证了能“懂语义”，又做到了能“对术语”，极大提升了整体覆盖率。
**怎么分词** 调用了jieba分词这个轻量级的中文分词库，同时遇到下划线和英文组合时会单独保留

### Q6.2：你提到把双路结果合并，其中 RRF（倒数秩融合）算法是怎么解释的？
**RRF 是什么**：
由于 FAISS 给出的通常是小数（比如余弦相似度 0.85），而 BM25 算出来的是一个无上限的绝对得分（比如 15.6），两把“尺子”的刻度完全不一样，所以绝对不能直接把他们的分数相加来算最终排名。
RRF（Reciprocal Rank Fusion）用来完美解决这个问题。它的核心思想是**完全抛弃原本的具体得分数值，只看“名次（Rank）”**。

**融合逻辑与实现**：
它的逻辑非常直白，得分公式就是 `1 / (该文档排名 + 一个常数)`。
举例说明：如果文档 A 在 BM25 排第一，而在 FAISS 那边拿到第5，那它总得分可能只是 1/6 左右。如果文档 B 恰好在两边都排到了第二，加上倒数算下来，文档 B 的总得分1/4就会超过文档 A，成为真正的第一名。这就保证了两方都认可的结果才最重要。

**防 HR 或面试官深究的“话术兜底”**：
1. **为什么选它，凭啥觉得它好？（工程导向解答）**：“在工业实践中，RRF 其实是性价比最高的 baseline 策略，因为它**防抖动能力极强，完全不需要依赖标注数据集去调校两路权重的比例**。如果真的用加权相加，换一批数据可能那个比例就不准了，RRF 开箱即用，特别契合我们希望快速建立可靠检索能力的敏捷落地需求。”
2. **如果在黑板上让你手撕，怎么写？**：“初始化一个空字典存储各文档最终得分；直接遍历两个 Top-N 的数组；拿着文档的内容或是独一无二的 ID 做 key，把基于排名的得分加在这个字典里；最后对字典里的 Value 做个倒序排列，输出对应的 Top 候选即可，时间复杂度极低。”
3. **如果有面试官挑刺问，常数（K）是多少？**：“业界和我在参考资料里看到的默认经验值通常设置为 60。目的是为了压掉‘一边排在极高位，另一边完全不相关’导致的畸形高分，保证长尾内容的平滑排序效果。”

### Q7.1：常见的RAG分片策略
固定长度切片（速度快，不依赖文档结构，但可能会破坏语义结构）
滑动窗口切片（保留一定上下文，但冗余数据多）
结构感知/标题分片（按照文档结构划分，需要结构化文档）
语义分片（相邻句子计算相似度，时间长）
层级分片（分为多个章节和段落，有点像目录，粗检索找到章节，细检索找到段落，也是复杂时间长）

### Q7.2：针对运维场景长文本（Log/Trace）的 RAG 方案是如何设计的？如何解决 Top-K 检索不准的问题？
**面试官视角考量：**
*这是高阶 RAG 岗/大模型应用的真实生产痛点。普通的文档还可以按长度切片，但 Log 日志和 Trace 链路这种“满篇全是时间戳和重复字符、只有几个真正错误码”的数据，用传统切片和余弦相似度检索会全军覆没（因为语义极弱，全是结构特征）。*

**标准回答（降维打击的话术策略）：**
首先，我的RAG系统是使用在运维知识库上面的，采用的就是BM25+FAISS 。。。

针对日志，由于我目前还没有接触过真实场景的运维日志，因此现在这个系统读取的日志都是我从github上找到的一些公开数据，我对这些数据做了针对性处理，只是设计了一个日志tools，提取出了时间和报错等字段，生成了结构化的数据返回给LLM。

但是我也思考过这个问题，运维的日志（Log/Trace）有一些与普通文档不同的特点：**比如它的有用信息密度比较少，噪声较多，原文非常长，时间戳、Info信息占据了大部分内容，而最重要的特征往往就是一行短短的 Error Stack 或者一个 Trace_ID 之类的**。如果直接用固定长度切片（比如 512 token 切一刀）然后去算余弦相似度，可能会出现检索极度不准、提取不到关键报错的问题。

我觉得主要可以分两步解决：
**第一步：日志不要原文入库进行 RAG，而是做“异常堆栈感知切分”与“关键摘要脱水（Distillation）”。**
- **预过滤/摘要**：针对几十万行的日志流，先把日志“脱水”，用本地部署的小模型，先把 `[INFO]` 之类的非关键信息剔除，提取出 “时间、服务名、TraceID、核心 Error 信息”，然后再将这段精简后的摘要去进行切分。
- **拒绝硬切片**：切分的时候不能简单按token切，有时报错堆栈会非常长非常复杂，如果按照token可能会从中间断开。可以使用带有正则匹配规则的切分器，保证整个多行错误块（Error Stack）作为一个完整的 Chunk 进入 RAG，避免被从中切断。

**第二步（检索层）：解决 Top-K 不准（双路召回 + 重排卡硬阈值）。**
- **BM25 精准捞词**：因为大量的运维报错是特定的类名或者 `trace_id`，缺乏语义，用混合检索里的 BM25 一定要在此时发生作用，靠强字面匹配捞回来。
- **重排卡绝对阈值**：如果前两步捞回来的 Top-K 分数不够（比如重排（Cross-Encoder）之后的 `sim_score`依然低于设定的阈值0.6），说明库里根本没切中相关日志，这时候宁可给大模型返回空块（让他回答“未找到相关信息”），也绝对不把一堆乱七八糟的弱相关日志塞进他的 Context 里，从而从根本上防治胡言乱语（幻觉）的出现。

### Q8：当模型生成了 tool_calls，到执行工具，再到把结果拼回到历史中去给模型再次推断，这个完整的数据结构流转（特别是 ToolMessage）是怎样的？（agent工具调用的底层机制，通用八股）
**标准回答（大白话梳理流程）**：
整体就是在一个带有记忆的对话列表里做追加：
1. **模型发出指令**：第一步，大模型判断需要用工具了，它会输出一个 `AIMessage`，但里面不一定是普通的文字，而是带有 `tool_calls` 字段（里面包含了要用什么工具、传什么参数，以及最重要的——**它会生成一个这把工具请求的独立 `tool_call_id`**）。
2. **执行并保存结果**：接着，由于我们设置了带工具的 Agent 节点，代码会自动去执行对应的 Python 函数（比如查日志）。查出来的结果，会被封装成一个特殊的数据结构叫做 `ToolMessage`。
3. **绑定的核心逻辑**：关键的一步就在这里，**新封装出来的 `ToolMessage` 必须强行带上第一步里生成的那个 `tool_call_id`**。这就像是取快递的凭证。
4. **重新发给大模型**：最后，把这个含有tool_call_id的 `ToolMessage` **追加** 到对话历史（消息列表）里，再把整个列表一起重新发给大模型。大模型一看到这个凭证对上了，就知道：“这是我刚才要求执行的工具结果，我已经拿到数据了”，于是它就会基于这个数据，生成最终的答案返回给用户。

**防深究话术：**
“其实这些底层的数据结构像 `AIMessage` 和 `ToolMessage` 都是 OpenAI 提出来的这套消息标准。我们在使用 LangChain 或者是 LangGraph 的时候，如果是标准流程，框架底层会自动用 `Node`（节点）帮我们处理这些拼装和 ID 对齐。我自己需要做的只是规范好 Python 工具函数（Tool）的出入参就行了。”

### Q9.1：多轮对话和上下文记忆是什么意思
多轮对话就是指，和大模型进行多轮沟通，大模型能记得之前沟通的内容。
比如历史沟通的对话就是上下文的一种，我们设计一个机制保持上下文，在同一会话内继续对话的时候，把上下文发送给大模型，这样大模型就仿佛拥有了记忆。
项目里我设计了分层的记忆管理机制。最简单的最近几轮对话，我们直接放在内存里，作为短期记忆，这能保证对话的连贯性。
当对话轮数增多，我们会把之前的对话内容进行处理，比如对前10轮历史对话进行总结摘要，作为长期记忆。
这样，模型就能同时记住最近聊了什么，以及过去聊的关键信息。
对历史对话做总结的目的，主要是为了避免太多的对话超过大模型的上下文窗口。
大模型上下文窗口就是一次对话中，能够记住和处理的信息总量的上限。说白了就是输入token+输出token的最大限制。（不同模型的上下文窗口大小不同，我看到的大多数常用模型的上下文窗口大小都是128k tokens）


### Q9.2：对话Agent 的记忆（Memory）是如何设计的/ Agent如何处理超出上下文窗口的长对话？
首先是定义了一个全局的状态字典，用于保存每个会话的内容，比如用户输入、LLM回复、工具返回之类的。然后我设计了一个三层的记忆管理机制
第一层是短期记忆，他会保留最近5轮对话的完整内容，这会被直接放入prompt中，保证对话的连贯性。
第二层是摘要或者说长期记忆，当对话轮数超过5轮时，每次对话前都会检查对话的token数量，如果超过模型的上下文窗口大小的70%，就会触发LLM压缩逻辑，保留关键信息，比如用户提到的核心问题、重要参数、已经解决的问题等，把长段文字压缩至大约4句话以内。

第三层是持久化记忆，每次触发压缩逻辑的时候，我们会把被压缩的文本

三记忆相互配合，既保证了短期对话的连贯性，又支持长期对话的信息检索，还避免了上下文窗口溢出的问题。

### Q9.3 具体是如何更新和维护的？
**直接回答**
按照对话Agent为例，我定义了一个全局的图状态字典ChatState，包含messages、rag_docs、answer三个字段，其中messages字段保存了所有的用户输入、LLM回复、tool返回信息、系统提示词，并且由于LangGraph内置了reducer（状态合并函数），它能自动把新消息追加并合并到messages列表中 
此外，光有 State 不行，因为 HTTP 请求是无状态的，每轮对话结束进程就销毁了，所以为了让下一轮对话还能记得上一轮的内容，在编译图结构的时候还要挂载一个 **大模型检查点Checkpointer** ，在每一轮对话结束后自动保存State（本项目使用的是MemorySaver，保存在内存中，而在实际生产中，也可能会用到redis等数据库来保存state）。
因此在多轮对话的时候只要把这个全局的state中的message发送给LLM就自然有了多轮对话能力。

### Q9.3 会话隔离怎么实现的
首先我定义了对话请求和响应的类，他们都包含一个session_id字段，这个session_id是会话初始化时后端生成的独一无二的id，每次前端调用后端对话接口（`\chat`或者`\chat\stream`）时都会带上session_id，在调用大模型的图服务时就会被封进参数中，即使有用户并发（从memorySaver或者RedisSaver中）读取历史会话，也只能看到自己的线程id对应的。
```python
class ChatRequest(BaseModel):
    message: str
    session_id: Optional[str] = None  # 不传则每次创建新会话

class ChatResponse(BaseModel):
    session_id: str
    answer: str
```

### Q11：什么是SSE，为什么不用websocket，为什么要用 SSE，你是怎么实现的？
**标准回答**：

前两个：
SSE和WebSocket都是实现实时通信的技术，但原理和适用场景不同。

先说说SSE吧，它（server-sent-event）是**基于HTTP协议**的，工作方式是，**客户端发起**一个普通的HTTP请求，但**服务器**不立即关闭连接，而是保持这个连接打开，并按照特定的格式(text/event-stream)**持续地、一段一段地向客户端推送数据。**

而WebSocket是一个独立的协议，它在初次连接时通过HTTP协议进行握手，握手成功后，连接就升级为WebSocket协议，之后双方就可以在这个连接上进行**全双工的双向通信**了，服务器可以随时发消息给客户端，客户端也可以随时发消息给服务器。

在选择时，我会考虑:如果需要双向实时交互，比如在线聊天、协同编辑，肯定选WebSocket。
如果只需要服务器向客户端推送实时数据，比如股票行情、通知、或者像我项目里的AI对话流，SSE就足够且更简单，因为它基于HTTP，不需要处理新的协议，后端实现和调试也方便一些。我们的场景主要是服务器向浏览器单向推送AI生成的文本，不需要双向通信，**用户体验就像是在看一个人实时打字**。
【SSE缺陷：单向通信、只能传输文本、并发连接有限制】
【
SSE在项目架构中的位置：

```
Browser
   ↓
HTTP POST /chat/stream
   ↓
Agent
   ↓
LLM
   ↓
SSE token stream
```
】

后两个：
由于整个诊断流可能包含多次 LLM 推理、本地检索加上外部 API 互调，一个接口返回可能长达 20-30 秒。如果用传统的 HTTP 短连接，用户体验极差，甚至可能触发浏览器超时丢弃连接。
使用SSE可以实施**流式回传**：
【实现细节在 `/chat/stream` 接口中：】
SSE使用标准的HTTP连接，客户端发起HTTP请求后，服务器返回的HTTP响应头部中 Content-Type: text/event-stream，并保持连接不关闭，后续发送消息也要按照SSE数据格式发送，也就是说每次（yield）传给前端的数据块必须以data:开头，并且以双换行符为结尾。而在消息的末尾还要传输done事件表示流式传输结束。

（连接不关闭的做法：使用FASTAPI的StreamingResponse，检测到生成器，HTTP就会变为分块传输，我创建了异步生成器，里面使用 yield 关键字， 可以让函数暂停执行并返回一个值，下次再继续从暂停的地方执行。每生成一个块就传给前端展示，直到done事件）
（1. 摒弃传统的 `invoke`，调用 LangGraph 的 `astream_events(version="v2")`。它能利用 Python 的异步迭代器，实时将底层任意节点运行状态的事件全都透传出来。）
（2. 为了只给前端展示最干净的‘打字机’文本，我过滤了 `event["event"] == "on_chat_model_stream"` 仅仅抓取底层大模型原生吐出的 token chunk。）
（3. 把模型推理产出的 token 块，通过 FastAPI 的 `StreamingResponse` 实时 yield [jiːld] 到前端侧，实现了类似 ChatGPT 的打字机渲染效果，极大抹平了等待焦虑感。）
